# GitHub 开源项目调研与可行性评估

## 执行摘要

这个方向并不是空白赛道，但仍然存在明确空档。

已经有很多开源项目覆盖了：

- AI coding agent
- 仓库打包与摄取
- 指令文件与规则标准

但仍然缺少一个真正面向团队的 **ContextOps 层**，也就是：

- 统一上下文格式
- 可组合的上下文片段
- 按任务装配上下文
- 多工具渲染
- 新鲜度和安全校验

因此，机会不在“再做一个 coding agent”，而在“做团队级上下文工程基础设施”。

## 现有项目分层

### 1. 执行层 agent

代表项目：

- `openai/codex`
- `cline/cline`
- `Aider-AI/aider`
- `OpenHands/OpenHands`

这些项目说明 AI 编码需求非常强，但重点在“执行”，不是“团队共享上下文治理”。

### 2. 仓库到上下文的打包层

代表项目：

- `yamadashy/repomix`
- `coderamp-labs/gitingest`

这些工具解决的是“怎么把仓库喂给模型”，不是“怎么把上下文作为团队资产长期管理”。

### 3. 规则与标准层

代表项目：

- `agentsmd/agents.md`
- `continuedev/rules`
- `ivawzh/agents-md`

这些项目说明跨工具 instruction 管理已经成为真实需求。

## 主要空白

当前还缺少：

- 一个跨主流 AI 工具的统一源格式
- 大型项目上下文的模块化与组合能力
- 面向 Bug、需求、评审、RCA 的任务级装配
- 文档过期后的自动提醒与治理
- owner、审批、优先级、作用域等团队协作机制

## 推荐方向

推荐把项目定义成：

**面向 AI 编码工具的 ContextOps 工具链**

推荐产品形态：

- CLI first
- monorepo
- 文档与代码同仓
- GitHub Action 集成

不建议第一阶段做：

- IDE 平台
- 云端产品
- 自主执行 agent runtime

## 三种产品方案

### 方案 A：只做规范

- 启动快
- 但很容易停留在理念层

### 方案 B：CLI first 的 ContextOps 工具链

- 工程价值最强
- 开源传播路径最好
- 推荐

### 方案 C：完整 AI 开发平台

- 叙事很大
- 竞争最激烈
- 当前不推荐

## ADR

### 背景

生态里已有很多相邻项目，但缺少团队级上下文基础设施。

### 决策

将产品定位在 ContextOps 层，而不是执行层 agent。

### 后果

- 差异化更清晰
- 避开最拥挤的红海

### 备选方案

- 通用 agent：被拒绝
- 纯规范项目：被拒绝
