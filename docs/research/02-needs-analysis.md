# 02 — 团队 AI 研发的真实需求分析

报告日期：2026-04-25
分析视角：用户/团队需求分析师
分析对象：用户原始想法 — 「在标准项目工程结构下增加标准的文本 md 工程，使 AI 能更好熟悉工程；文本工程版本管理、团队共享；不熟悉工程的人只需专注自身任务」

---

## 1. 痛点拆解

从用户原始想法中提取 7 个独立、可验证的痛点：

**P1：仓库知识以「部落知识」形式存在，AI 无法消费**
验证方式：让一个 AI agent 在无文档的大型仓库中修复跨模块 bug，统计 tool call 次数和正确率。
Meta 在 4100+ 文件的数据管线上实测：AI agent 在无上下文文件时"got lost fast — missing naming conventions, breaking serialization compatibility, producing code that compiled but was subtly wrong"。生成 59 份上下文文件后，tool call 减少 40%。

**P2：多 AI 工具的入口文件分裂，团队手工同步**
验证方式：统计一个团队维护 CLAUDE.md / AGENTS.md / .cursor/rules / copilot-instructions.md 时的重复内容比例。
2026 年 3 月对 234 个开发团队的分析显示，67% 的团队在多个配置文件间维护重复内容。

**P3：静态文档无法按任务类型提供最小必要上下文**
验证方式：对比全量上下文 vs 按任务装配上下文下 AI 输出的聚焦度和 token 消耗。
AI agent 在 50K+ 行代码库上会用掉 80-90% 上下文窗口仅用于加载文件，几乎没有空间留给推理。Coding agent 花费高达 60% 的时间搜索上下文而非解决问题。

**P4：文档过期（Context Rot）产生稳定误导**
验证方式：标记一批文档的最后更新时间，对比过期文档指导下的 AI 输出错误率。
Chroma Research 的实验表明，随着上下文中过期/不相关信息增加，LLM 性能持续下降。Anthropic 2026 报告确认维护 CLAUDE.md 的团队"坏建议会话"减少 40%，反证不维护时问题严重。

**P5：新人 / 外部贡献者 onboarding 成本高，AI 无法补偿知识缺口**
验证方式：测量新人在有/无结构化上下文文件时完成首个 PR 的时间。
Meta 实测：原本需要约 2 天的复杂工作流调研，在有上下文文件后压缩到约 30 分钟。Anthropic 报告将 CLAUDE.md 类比为"给 agent 的 onboarding 文档"。

**P6：上下文文件缺乏版本管理和审核流程**
验证方式：检查团队的 CLAUDE.md 是否在 git 中、是否经过 PR review、是否有 owner 分配。
ContextArch 分析指出团队的默认反应是"每当 agent 犯错就加一条规则，但很少删除规则"，导致文件膨胀、自相矛盾。只有 32% 的工程组织有带强制力的治理政策。

**P7：AI 加速产出但同步引入不稳定性，缺乏质量护栏**
验证方式：对比引入 AI 前后的 PR 变更失败率和事故率。
Cortex 2026 基准报告：PR 数量每作者增长 20%，但事故率增长 23.5%，变更失败率增长约 30%。**速度提升伴随质量下降是可量化的。**

| 痛点 | 被讨论频率 | 是否有量化证据 | 典型引用 |
|---|---|---|---|

| P2 入口分裂 | 高 | 67% 团队重复维护 | ContextArch 2026 |

## 3. 目标用户画像

### 3.1 谁最痛？（按痛感排序）

1. **中大型仓库的平台/基础设施团队** — 最痛。他们负责跨 50+ 仓库统一规则，手工同步成本最高，且是 AI 工具的治理责任方。Cortex 报告显示正是这些团队承受着 +30% 变更失败率的压力。
2. **中大型仓库中需要跨模块工作的开发者** — 痛感强。新需求或 bug 常跨 order/inventory/payment 多个模块，AI 在无上下文时表现最差的场景就是跨模块理解。
3. **开源项目维护者** — 中等偏高。外部贡献者不了解项目约定，PR 质量参差不齐。AGENTS.md 被 60,000+ 仓库采用说明了这个需求的规模。
4. **新人/外包/轮岗人员** — 痛但无决策权。他们是受益者但不是采购者/贡献者。
5. **个人开发者** — 轻微。项目知识在自己脑中，不需要外化。

### 3.2 谁最愿意贡献？

- **平台团队的 DevEx 工程师**：他们的 KPI 就是降低全团队的摩擦，有动力贡献工具
- **开源项目维护者**：已经在写 CONTRIBUTING.md，有维护文档习惯
- **Context Engineering 布道者**：2026 年 context engineering 被 Anthropic 列为"年度最重要技能"，有一批人在积极传播

### 3.3 谁是伪用户？

- **「用 AI 写个人项目」的独立开发者**：看起来会用，但项目规模不够大，脑中已有全部上下文，不需要文本工程
- **「希望 AI 全自动写代码」的管理者**：期望与产品定位不匹配
- **小团队 (2-3 人)**：沟通成本低，口口相传够用，文本工程的 ROI 为负

## 4. 典型场景剧本

### 场景 A：新人入职 Java Spring 大型单体仓库，修跨 order/inventory 的 bug

**没有文本工程时**：新人张三入职第 2 周收到一个 bug：下单后库存未正确扣减。他打开 Claude Code，输入 bug 描述。Claude 建议修改 OrderService.java 中的 decrementStock 调用，但这个方法在半年前已被废弃，真正的扣减逻辑走的是 InventoryClient 的异步消息。Claude 不知道这个架构变更（部落知识），新人花了 2 天才在 code review 中被告知方向错误。

**有文本工程后**：仓库中 `docs/ai/modules/order.md` 记录了 order-inventory 的调用链变更历史，`playbooks/bugfix.md` 定义了跨模块 bug 的排查步骤。执行 `ctx compose --task bugfix --path services/order --diff HEAD~5..HEAD` 后，Claude 收到精准的模块间依赖说明，直接定位到 InventoryClient.publishDecrementEvent() 的消息格式不匹配。**从 2 天缩短到 2 小时。**

**相对竞品的额外价值**：rulesync 只做规则同步，不做任务级装配；agents-md 做片段组合但不做 diff-aware 的按需筛选；Packmind 做治理但不做最小上下文组装。`ctx compose --task --path --diff` 是无人覆盖的组合。

### 场景 B：外部贡献者给开源项目提 feature

**没有文本工程时**：贡献者李四要给一个 3 万行的 Rust CLI 工具增加 JSON 输出格式。他读了 README 和 CONTRIBUTING.md，用 Claude 生成了代码，但不知道项目有严格的 error-handling 约定（所有错误必须走 `miette` 而非 `anyhow`）。PR 被要求重写，贡献者流失。

**有文本工程后**：项目根目录的 AGENTS.md 由 ctxops 渲染，自动包含了 error-handling 约定、模块边界、命名规范。贡献者的 AI 工具读取后生成的代码自动遵循 `miette` 约定。**PR 一次通过 review 的概率大幅提升。**

**相对竞品的额外价值**：Packmind 偏向团队内部治理，对匿名外部贡献者的场景支持弱；rulesync 能做多工具同步但不维护语义上下文（只同步规则，不提供架构知识）。

### 场景 C：平台团队在 50 个 repo 间统一 AI 规则

**没有文本工程时**：DevEx 团队的王五需要在 50 个微服务仓库中统一 "不准使用 System.exit()" 这条规则。他在每个仓库的 CLAUDE.md 中手动添加，3 个月后有 15 个仓库的规则已过期（被覆盖或被删除），且无人知道。

**有文本工程后**：在 monorepo 或 central config repo 中维护 `project-baseline.md` 片段，各仓库通过 `imports: [project-baseline]` 引用。`ctx doctor` 在 CI 中检测到引用断裂或规则过期时报红。**变更一处，50 个仓库同步生效。**

**相对竞品的额外价值**：rulesync 做多工具同步但不做 import 图和循环检测；Packmind 做跨仓库治理但是商业产品（核心治理功能在企业版）。ctxops 提供开源、可自托管、带 import 图校验的跨仓库方案。

## 5. 「不需要熟悉工程底层」的可达性分析

用户的理想目标：「不熟悉的人在使用 AI 开发时，不需要过多了解工程底层或熟悉工程，只需要专注于自身的任务」。

### 5.1 三个结构性约束

**约束一：Context Rot（上下文腐烂）**
文档必然落后于代码。Meta 的方案需要"每几周自动化验证文件路径、检测覆盖缺口、重跑质量检查器"。如果没有持续维护机制，文本工程本身会成为误导源。**可达的子目标是：建立自动化的 freshness 检测（`ctx doctor`），而非假设文档永远正确。**

**约束二：Token Budget（令牌预算）**
即使 2026 年 Claude 和 Gemini 都支持 1M token 上下文窗口，但 Chroma Research 证明随着输入 token 增加，LLM 性能下降（context rot 效应）。**更多上下文不等于更好表现。** 可达的子目标是：按任务装配最小必要上下文（`ctx compose`），而非把所有文档塞给 AI。

**约束三：AI 检索能力的天花板**
AI agent 在大型代码库中仍然花费 60% 的时间搜索上下文。即使有完美文档，模型仍可能"看到了但没用上"（注意力稀释）。HN 上的高赞评论直言："very large context windows aren't useful in practice"。**可达的子目标是：提供精准的导航（"compass, not encyclopedia"），而非百科全书。**

### 5.2 可达性评估矩阵

| 子目标 | 可达性 | 前提条件 |
|---|---|---|
| 新人不需要了解全部代码即可修 bug | **高** | 模块文档 + playbook + compose 装配 |
| 新人不需要了解任何工程知识即可工作 | **低** | AI 无法替代对业务域的基本理解 |
| 外部贡献者 PR 一次通过率 > 80% | **中** | 需要高质量 AGENTS.md + 规则覆盖率 |
| 完全不读代码只看文档 | **极低** | 违反软件工程基本规律 |
| 文档自动保持最新 | **中** | 需要 CI 集成 + freshness 检测 + 人工兜底 |
| 跨模块理解无需口口相传 | **中高** | 模块间依赖文档 + 变更历史 + ADR |

### 5.3 结论

**理想态距离：60% 可达。**

文本工程可以将"必须深度熟悉工程"降级为"只需理解业务域 + 阅读 AI 组装后的精简上下文"，但无法实现"零了解即可工作"。最有价值的可达子目标是：

1. 将跨模块 bug 排查从"2 天 + 老人指导"压缩到"2 小时 + AI 辅助"（Meta 已验证）
2. 将外部贡献者 PR 的返工率从 60%+ 降到 30% 以下
3. 将文档过期导致的 AI 误导从"沉默失败"变为"CI 显式报红"

不可达的部分需要接受：AI 不能替代对业务逻辑的基本理解，也不能在文档缺失时凭空补全组织知识。

---

## 总结

**5 个最强痛点（排序）**：
1. **P1 — 部落知识不可被 AI 消费**（Meta tool call -40%）
2. **P4 — 文档过期产生稳定误导**（Chroma + Anthropic 双重验证）
3. **P7 — AI 加速产出但质量下降**（Cortex 数据：事故率 +23.5%，CFR +30%）
4. **P3 — 无任务级最小上下文装配**（agent 60% 时间找上下文，且是唯一差异化点）
5. **P2 — 多工具入口分裂**（67% 团队重复维护）

**目标用户排序**：平台团队 > 跨模块开发者 > 开源维护者 > 新人/外包 > 个人开发者

**「不需熟悉工程底层」可达性**：60%。可以将"深度熟悉"降级为"基本业务理解 + AI 辅助精简上下文"，无法零了解即工作。
